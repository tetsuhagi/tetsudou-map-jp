# 駅命名・命名ID ガイド

`data/stations.csv` に駅を追加するときの判断基準。私鉄が増えると
「同一場所に複数会社の駅がある」「同名で別の駅がある」ケースが出てくる
ので、命名ルールを明文化しておく。

新しい路線（特に私鉄）を追加するときに **必ず一度ざっと読む** こと。

---

## 結論（迷ったらこれだけ守る）

1. **駅IDはローマ字大文字、`_` 区切り、アルファベットのみ**
   - 例: `NAGOYA`, `MEITETSU_NAGOYA`, `KEISEI_UENO`
2. **表示名は実在する正式駅名をそのまま使う**
   - 例: 「名古屋」「名鉄名古屋」「京成上野」「上野」
3. **物理的に同じ駅（改札連絡可・同一駅構内）なら 1 ID に統合**
   - 例: 金山（JR/名鉄共用）, 日暮里（JR/京成共用）, 関西空港（JR/南海共用）
4. **物理的に別駅で正式駅名が違うなら別 ID**
   - 例: 名古屋 / 名鉄名古屋, 上野 / 京成上野

## 1. 駅ID（`station_id`）の命名

### 1-1. 形式

- **アルファベット大文字 + `_` 区切り**
- 数字・小文字・ハイフンは使わない
- ローマ字は **訓令式かヘボン式かどちらかに統一**（実用上はヘボン式中心）

OK例:
```
NAGOYA, KEISEI_UENO, MEITETSU_ICHINOMIYA, SHIN_IMAMIYA, CENTRAIR
```

NG例:
```
nagoya          ← 小文字
keisei-ueno     ← ハイフン
nagoya2         ← 数字
名古屋          ← 日本語
```

### 1-2. プレフィックス（運営社名）の付け方

| パターン | プレフィックス | 例 |
|---|---|---|
| JR系・最初に登録した路線が使う駅 | **なし** | `NAGOYA`, `UENO`, `OSAKA`, `KYOTO` |
| 同名駅で別運営者の駅を追加する場合 | **運営社略称_** | `MEITETSU_NAGOYA`, `KEISEI_UENO`, `KINTETSU_NAMBA`(将来) |
| 駅名にもともと運営社が入っている場合 | **運営社略称_** + 残り | `MEITETSU_GIFU`（正式駅名「名鉄岐阜」）|

### 1-3. 採用済みプレフィックス

| プレフィックス | 運営社 | 由来 |
|---|---|---|
| `MEITETSU_` | 名古屋鉄道（名鉄） | 名鉄岐阜 / 名鉄一宮 / 名鉄名古屋 等の正式駅名 |
| `KEISEI_` | 京成電鉄 | 京成上野 等の正式駅名 |

## 2. 表示名（`name` 列）のルール

### 2-1. 基本

**駅の正式名称をそのまま使う**。Wikipedia や運営社公式サイトに記載の
駅名がベース。

```
station_id            name
NAGOYA                名古屋        ← JR名古屋駅の正式駅名「名古屋」
MEITETSU_NAGOYA       名鉄名古屋    ← 名鉄の正式駅名「名鉄名古屋」
UENO                  上野          ← JR上野駅の正式駅名「上野」
KEISEI_UENO           京成上野      ← 京成の正式駅名「京成上野」
```

正式駅名にすでに運営社名が含まれている（例：「名鉄名古屋」「京成上野」）
場合は、それをそのまま使えばよい。

### 2-2. 例外的な共用駅

物理的に同一駅で複数社が乗り入れている場合、**駅名は1つに統合**：

```
NIPPORI               日暮里        ← JR+京成、ホームは別だが同一駅構内
KANAYAMA              金山          ← JR+名鉄、同一駅ビル・改札内連絡
KANSAI_AIRPORT        関西空港      ← JR+南海、ホーム共用
SHIN_IMAMIYA          新今宮        ← JR+南海、隣接ホーム
```

これらは正式駅名がすべて「日暮里」「金山」など共通（プレフィックスなし）
なので、迷うことなく単独 ID で OK。

## 3. 判断フロー（新しい駅を追加するとき）

```
            新規駅を追加したい
                  │
                  ▼
        既存に同名駅がある？
              │     │
            なし   あり
              │     │
              ▼     ▼
   そのままIDを        既存駅と物理的に
   作る（プレフィックス  同じ場所か？
   なし）               │      │
                       はい    いいえ
                        │      │
                        ▼      ▼
                  既存ID使用    新IDを作る
                  （統合）       （プレフィックス付き）

                                ※「物理的に同じ」とは:
                                  - 同一駅ビル
                                  - 改札内・改札外連絡通路で繋がっている
                                  - 徒歩2分以内の隣接構内
                                を目安にする
```

### 3-1. 具体例での判断

| ケース | 判断 | 結果 |
|---|---|---|
| 名鉄ミュースカイ追加で「名古屋」必要 → 既存に NAGOYA (JR) あり、ただし名鉄名古屋は **別の改札・別のホーム** | 別駅扱い | 新 ID `MEITETSU_NAGOYA` 追加 |
| スカイライナーで「上野」必要 → 既存に UENO (JR) あり、ただし京成上野は **JR上野駅から南へ約400m**、別建物 | 別駅扱い | 新 ID `KEISEI_UENO` 追加 |
| 京成スカイライナーで「日暮里」必要 → 既存に NIPPORI なし、JR/京成は **同一駅構内・改札内連絡** | 同一駅 | 新 ID `NIPPORI` 追加（プレフィックスなし）|
| 南海ラピートで「関西空港」必要 → 既存に KANSAI_AIRPORT (HARUKA で使用) あり、JR/南海は **ホーム共用** | 同一駅 | 既存 `KANSAI_AIRPORT` を流用 |

## 4. マップ表示時の重複（既知の課題）

`MEITETSU_NAGOYA` と `NAGOYA` は座標が **約25m差** で、地図上ではほぼ
完全に重なる。これは現実そのもの（同じ駅ビルに2社の駅が共存）なので、
**意図的に維持**。

| 検討した対応 | 結論 |
|---|---|
| 座標をズラす | 不自然なので不採用 |
| 駅マーカーをクラスタリング | UI拡張時に検討。現状は実害なし |
| 1 ID に統合する | 名鉄名古屋というアイデンティティが消えるので不採用 |

将来、駅マーカーをクリックして詳細表示する UI を作るときに
クラスタリングを検討する。それまでは現状維持。

## 5. 商標・商号に関する整理

「JR」「名鉄」「京成」「南海」などの事業者略称・商号を駅名に使うことに
ついて、**法的リスクは極めて低い**。

### 5-1. 理由

- **事実の参照**: 「名鉄名古屋」は名古屋鉄道が定めた正式駅名であり、駅名を表示することは事実の参照
- **業界標準**: Google マップ・乗換案内・ナビタイム・各種鉄道メディア・個人ブログすべてで普通に使用
- **既存の免責**: サイトフッター・トップページの info モーダルに以下を明示済み

> 本サイト上に表示される列車名・路線名は、各鉄道事業者の登録商標または
> 商号です。本サイトでは事実の参照として表示しているものであり、各事業者
> との提携・関連はありません。

駅名併記もこの免責文言のカバー範囲内。

### 5-2. リスクが上がる線引き

下記は **やらない**:

- ロゴマーク・社章・コーポレートシンボルの転載
- 「JR公式」「名鉄推奨」など提携を装う表記
- 独自フォント・ブランドカラーをそのまま再現
- 「名鉄ナビ」のような商号と紛らわしいサイト名・サービス名
- 「JR」「名鉄」を**商品・サービス名の一部として宣伝的に**使う

下記は **OK**:

- ✅ 「名鉄名古屋」「JR名古屋」のような駅名併記
- ✅ 「JR西日本」「JR東海」と運行会社名を事実として表示
- ✅ 「名鉄2000系」など車両型番の参照
- ✅ 「名古屋鉄道」「京成電鉄」の社名を運営者として記載
